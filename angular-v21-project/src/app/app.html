3.4 Angular 21 vs Angular 16 — Key Differences Standalone default · esbuild · Signals · @if/@for · Vitest · no app.module.ts

The evolution from Angular 16 to Angular 21 introduced six major shifts: Standalone components became the default (no NgModule required); the build system moved from Webpack → esbuild (significantly faster builds); Signals became the recommended reactive primitive over BehaviorSubject; built-in control flow (@if, @for, @switch) replaced structural directives (*ngIf, *ngFor); Vitest replaced Karma/Jasmine; and Signal Forms provide a new approach alongside Reactive Forms. In Angular 19+, standalone: true in the @Component decorator is fully optional — it is the implicit default.

FeatureAngular ≤ 16Angular 17–21
Module systemNgModule mandatoryStandalone default — NgModule optional
BootstrappingbootstrapModule(AppModule)bootstrapApplication(App, appConfig)
Build systemWebpackesbuild (much faster)
Control flow*ngIf, *ngFor@if, @for, @switch
Reactive stateRxJS / BehaviorSubjectSignals (stable, recommended)
Test runnerKarma + JasmineVitest (Karma deprecated)
3.5 Standalone Components & Bootstrapping No app.module.ts · bootstrapApplication() · app.config.ts providers · component-level imports[]

Standalone components declare their own dependencies in imports: [...] inside @Component — there is no app.module.ts in this project. The two signals of a standalone component are: (1) the imports: [] array is present, and (2) from Angular 19+, standalone: true is implicit and can be omitted.

Bootstrapping changed from platformBrowserDynamic().bootstrapModule(AppModule) (Angular 16) to bootstrapApplication(App, appConfig) (Angular 17+). app.config.ts now holds all providers — provideRouter(), provideHttpClient(), etc. — that were previously split across NgModule.imports and NgModule.providers. Route params bind automatically to input() signals via withComponentInputBinding() registered in this file.

3.6 Angular Lifecycle Hooks ngOnInit · ngOnChanges · ngAfterViewInit · ngOnDestroy — full hook order

Every component goes through a predictable lifecycle. The constructor is a TypeScript/OOP concept — not a lifecycle hook. Angular hooks are interfaces the component implements.

Full order: constructor → ngOnChanges (before init, if @Input present) → ngOnInit (once, after first ngOnChanges) → ngDoCheck (every CD cycle — avoid heavy logic here) → ngAfterContentInit (once, after <ng-content> projected) → ngAfterContentChecked → ngAfterViewInit (once, after component view rendered) → ngAfterViewChecked → ngOnDestroy (just before removal).

The 3 you'll use 95% of the time: ngOnInit — API calls, subscriptions, initial setup; ngOnChanges — react to @Input changes from parent; ngOnDestroy — unsubscribe observables, clear intervals (prevent memory leaks). ngAfterViewInit is needed when accessing @ViewChild references.

The Data Binding Demo (Topic 3.1) already implements all these hooks live — open the browser console to see each hook fire in sequence as the component mounts.

3.7 Built-in Control Flow — @if / @for / @switch Angular 17+ replaces *ngIf / *ngFor · mandatory track · @empty block

Structural directives (*ngIf, *ngFor, *ngSwitch) change the DOM structure by adding or removing elements. Angular 17+ replaces all three with built-in control flow blocks — no import of NgIf, NgFor, or CommonModule needed.

Conditional: Angular 16 used *ngIf="condition; else tmpl" with <ng-template #tmpl>. Angular 17+ uses @if (condition) { ... } @else { ... } — reads like plain TypeScript.

List rendering: Angular 16 used *ngFor="let x of list; trackBy: fn". Angular 17+ uses @for (item of items; track item.id) { ... }. The track expression is mandatory in @for — it is the equivalent of trackBy and ensures Angular only re-renders changed DOM nodes (critical for list performance). An @empty { ... } block renders when the list is empty.

Why track matters: Without it, Angular destroys and recreates all DOM nodes on every change-detection cycle — even if only one item changed. With track item.id, only the changed/added/removed nodes are touched. The Employee Table (Topic 3.3) uses @for with track in practice.

3.8 Dependency Injection — inject() vs Constructor Angular 17+ functional DI · providedIn: 'root' singleton · tree-shakeable services

Dependency Injection is Angular's mechanism for providing a service instance to a component without the component creating it directly. This decouples components from service implementation.

Angular 16 — constructor injection: constructor(private svc: EmployeeService) {} — the service is listed as a constructor parameter; TypeScript's reflection metadata informs Angular what to inject.

Angular 17+ — inject() function (preferred): private svc = inject(EmployeeService); — declared as a class field initialiser. No constructor needed, works cleanly with standalone components, and is more compositional (you can call inject() in utility functions as long as you're in an injection context).

providedIn: 'root' on the @Injectable decorator creates a singleton — one shared instance across the entire app. It is also tree-shakeable: if no component injects the service it is excluded from the production bundle automatically. The EmployeeService in this project uses this pattern — open src/app/services/employee.service.ts to see it live.

5.1 Core Routing — SPA Navigation RouterOutlet · RouterLink · RouterLinkActive · redirect / wildcard / child routes

Angular Router maps URLs to components without full page reloads. <router-outlet> is the placeholder where matched components render. routerLink replaces href for client-side navigation. routerLinkActive adds a CSS class when the route is active. Route types: redirect (pathMatch: 'full'), specific, wildcard (**), and child routes. The Courses route uses child routes — its own nested <router-outlet> renders Overview and Details.

▶ Try it: 🏠 Home 👥 About 📚 Courses (child routes)
5.2 Lazy Loading — loadComponent() Dynamic import() · separate JS chunks fetched on first navigation · smaller initial bundle

Lazy loading defers the download of a component's JS chunk until the user actually navigates there. loadComponent: () => import('./path').then(m => m.MyComponent) is a dynamic import — the module is fetched on demand, not bundled at startup. Enroll is the classic example: only 1 in 10 users clicks it, so it shouldn't be in the initial bundle. In the browser Network tab you'll see a new chunk appear when you first visit a lazy route.

▶ Try it: 🎓 Enroll lazy 🛒 Products lazy
5.3 Route Guards — canActivate Functional CanActivateFn · block unauthenticated access · returnUrl redirect

A canActivate guard runs before a route is entered. The functional CanActivateFn (Angular 17+) uses inject() — no class needed. Dashboard is protected: unauthenticated users are redirected to /login?returnUrl=/dashboard. After login, the Login component reads returnUrl from query params and navigates there automatically. Guard returns true to allow, false to block, or a UrlTree to redirect.

▶ Try it: 🔐 Dashboard (guarded) 🔑 Login (admin / 1234)
5.4 Route Guards — canDeactivate Functional CanDeactivateFn · unsaved-changes protection · form.dirty check

canDeactivate runs before the user navigates away from a route. The guard calls component.hasUnsavedChanges() — if the method returns true, the browser shows a confirmation dialog. This prevents accidental loss of form data. The component implements a HasUnsavedChanges interface; the guard is a plain CanDeactivateFn<HasUnsavedChanges> function — no class, no decorator needed.

▶ Try it: 🚧 canDeactivate Demo — type, then navigate away
5.5 Route Parameters & withComponentInputBinding() :id segment auto-bound to input() signal · computed() lookup · no ActivatedRoute

Route params embed data directly in the URL (/products/2). With withComponentInputBinding() enabled in app.config.ts, the :id segment is automatically bound to a matching input<string>('id') on the component — no ActivatedRoute subscription needed. A computed() signal then looks up the product in the static array and re-runs whenever the param signal changes.

▶ Try it: 🛒 Product List → click a product for detail
5.6 Reactive Forms — FormBuilder, FormGroup, Validators Typed controls · state flags (.dirty / .touched) · patchValue / setValue · service-backed flow

Reactive Forms define the form model in TypeScript. inject(FormBuilder) creates FormGroup with typed FormControls and Validators. Errors show only after the user has touched a field (not on load). patchValue() updates specific fields; setValue() requires all fields. getRawValue() returns a typed object including disabled controls. The Register → Summary flow demonstrates sharing data across routes via a singleton RegistrationDataService (no HTTP needed).

▶ Try it: 🧾 Registration Form 📄 Summary (post-submit)
5.7 FormArray — Dynamic Repeated Fields Variable-length field groups · push() / removeAt() · formArrayName + [formGroupName]

FormArray manages a variable number of identical field groups — e.g. multiple phone numbers or skills. Use formArrayName on the container, then [formGroupName]="i" on each child (by $index). push() adds a new group; removeAt(index) removes one. A typed getter avoids casts in the template: get phones() { return this.form.get('phones') as FormArray; }

▶ Try it: 🧩 FormArray Demo (phones + skills)
5.8 FormControl Advanced & Signal Forms (Angular 21) nonNullable · toSignal vs computed() · experimental signal-based forms without FormControl

{ nonNullable: true } on a FormControl changes reset() to restore the initial value instead of setting null. Never use computed() to read form.valid / form.dirty — those are plain class getters, not signals, so computed() won't re-run. Use toSignal(form.events.pipe(map(...))) instead. Signal Forms (Angular 21, experimental) replace FormControl entirely with plain signal() state and computed() for derived validity — no ReactiveFormsModule needed.
